the two state machines.

types, App.vue lighter, two deferred-items entries retired.

## Approach

### The composable shape (post mid-execution flip)

The plan envisioned the composable owning the modal ref:
`useDirtyBoardGuard()` returning
`{ confirmLoadModalRef, handleLoadCard }`. During execution this
ran into a vue-tsc + Vue 3 typing interaction:

- The string ref form (`<ConfirmLoadModal ref="confirmLoadModalRef" />`)
  works only when `confirmLoadModalRef` is a top-level local in
  the component's `<script setup>`. Destructuring from a
  composable's return — `const { confirmLoadModalRef, … } = …()` —
  binds the name locally, but vue-tsc's `noUnusedLocals` check
  flags it as unread, because the string-ref lookup is invisible
  to TypeScript.
- The bound `:ref="confirmLoadModalRef"` form expects a
  `VNodeRef | undefined` per Vue's typing, but a
  `Ref<InstanceType<typeof ConfirmLoadModal> | null>` doesn't fit
  that shape (the `null` half mismatches).

The fix: **flip the composable's signature to take the ref as a
parameter.** App.vue declares the ref at top level (where vue-tsc
tracks it through string-ref usage); the composable consumes it.
This mirrors `useScopedScroll.ts`'s parameter-style — the closest
sibling pattern in the codebase.

The flip also has an architectural-honesty gain: the ref's
lifecycle is genuinely the template's, not the composable's. The
composable consumes a refs-shaped capability rather than owning
something it can't actually own.

Final API:

```typescript
export function useDirtyBoardGuard(
  confirmLoadModalRef: Ref<InstanceType<typeof ConfirmLoadModal> | null>,
): { handleLoadCard: (card: ReviewCard) => Promise<void> }
```

### Behavioral changes vs. preservation

**Preserved** exactly:
- The three-branch policy (`ask` / `new` / `overwrite`).
- The remember-the-choice persistence via `updateRegistry`.
- The SGF parse + board create + navigate flow.
- The board-id mutation pattern.

**Changed** (deliberately, per plan):
- The silent early-return at App.vue:109 became an explicit
  fail-loud `throw` if `confirmLoadModalRef.value` is null at
  handler-call time. Per ADR-0002, the contract violation
  surfaces audibly rather than silently.
- The string-parsing of `'-saved'` suffix is gone. Consumed via
  structured `{ action, remember }` destructuring.

### `ConfirmLoadModal.vue` — type cleanup + ADR-0006 retrofit

- `LoadAction` stays narrow (`'new' | 'overwrite' | 'cancel'`).
- New exported `LoadResult` interface: `{ action, remember }`.
  Both fields are `readonly`.
- `open(): Promise<LoadResult>` instead of `Promise<LoadAction>`.
- `handle(action)` emits the structured object; the
  `as LoadAction` cast is gone.
- Header retrofit per ADR-0006: HTML comment header at the top
  of the SFC removed; equivalent JSDoc moved to top of
  `<script setup>`.

### Edits to `frontend/src/App.vue`

- Removed the cluster (line 97 ref + lines 99–143 handler).
- Added composable import and one setup line (with the
  parameter-flip shape):

  const { handleLoadCard } = useDirtyBoardGuard(confirmLoadModalRef);

  (removed from the store import). All confirmed unused via grep

## Lessons (mid-execution)

The composable signature flip is the headline lesson of C2.2:

> When extracting a cluster that owns a template ref, the cleanest
> shape is composable-takes-ref-as-parameter, not
> composable-owns-and-returns-ref. The former works with vue-tsc's
> string-ref tracking and matches the `useScopedScroll` precedent;
> the latter fights both vue-tsc and Vue's `:ref` typing.

This was not anticipated in the plan; it surfaced when the build
flagged `confirmLoadModalRef` as unread despite being template-
bound. The fix took ~5 minutes and produced a cleaner shape than
the original plan; capturing it here so future C2.x extractions
that touch refs (e.g., a hypothetical `useMintCardFlow`) start
with the parameter-style by default.
