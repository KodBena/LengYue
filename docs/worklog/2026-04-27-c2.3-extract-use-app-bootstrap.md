## Context

C2.2 reduced App.vue from 569 to 520 lines by extracting
`useDirtyBoardGuard`. C2.3 is the third extraction — the
app-bootstrap cluster at App.vue:119–136: a `SyncService`
instance creation followed by an `onMounted(async)` running the
cold-start sequence (auth-first auto-login, sync hydration,
resource preloading, tag-dictionary fetch).

One important wrinkle confirmed by Phase 1: the `sync` instance is
also referenced from the template at App.vue:370 — the Settings
tab's "Force Persistence" button binds `@click="sync.forceSave()"`.
The composable cannot be pure side-effect; it must own the
`SyncService` instance and expose it to the caller for the
template binding to keep working.

## Approach — applying the C2.2 lesson

The C2.2 worklog captured: **for clusters that own a shared
dependency, take the dependency as a parameter; own only what the
composable genuinely creates.** Applied to C2.3:

- **`auth` is shared** with the rest of App.vue (the template binds
  `auth.isAuthenticated`, `auth.username` etc. via `<UserBadge />`).
  The composable receives `auth` via parameter, observing the same
  reactive instance the App-level `useAuth()` call produced.
- **`sync` is genuinely owned** by the bootstrap. The composable
  creates it and exposes it in the return shape.

Final API:

```typescript
export function useAppBootstrap(
  auth: ReturnType<typeof useAuth>,
): { sync: SyncService }
```

Behavior preservation: exact. The composable's `onMounted` body is
the verbatim cluster body, including the auth-first ordering
comment (the comment IS the reason — kept verbatim) and the
fail-soft `console.warn` on tag-fetch failure (per ADR-0002's
"console.warn is appropriate when 'this shouldn't happen, but if
it does, the rest of the system can continue'").

## Bounded-stopping evaluation — closing the C2 arc

The user-agreed scope for C2 was definition (a): extract the seams
that yield clean composable shapes; stop when the remaining App.vue
is wiring/template/minimal logic. **C2.3 fires that condition.**

Remaining clusters in App.vue (post-C2.3) per the original
decomposition:

| Cluster | Lines | Verdict |
|---|---|---|
| Mint Flow (mintModalRef + triggerMint) | ~5 | Thin; not worth extracting. |
| Komi handler | ~9 | Thin; not worth extracting. |
| Move Counter (moveNumber computed) | ~22 | Thin; pure read. |
| Board Move Handler | ~12 | Thin; routes to review-session vs mutation. |
| Node Select Handler | ~8 | Thin; pure routing. |
| Registry Update Handlers (3 wrappers) | ~3 | Thin; pure passthrough. |
| Visits Override | ~5 | Thin adapter. |
| Intermission Chart (intermissionSeries computed) | ~7 | Thin; pure derivation. |
| Ponder Watcher (`watch` on currentNodeId) | ~14 | Could extract; modest payoff. |

Total remaining "extractable" clusters: ~85 lines if all extracted,
which would land App.vue at ~415 lines — still well over ADR-0007's
soft target. And per ADR-0007: *"never compress logic to fit a
budget — that's code golf and inflates working-memory cost per
line."* Extracting these thin clusters would dilute App.vue's
coherence (each composable carries import overhead, JSDoc, and
parameter-passing churn) without making any of them
individually clearer.

The remaining App.vue is roughly:
- Imports + composable wiring (~50 lines).
- Computed values + thin handlers (~80 lines).
- Template (~260 lines).
- Styles (~110 lines).

The bulk is template + styles, not logic. ADR-0007's content-aware
contraction posture acknowledges this: *"Format reflects edit
cadence. Content rarely hand-edited contracts to maximize the
visible budget for content that is."* If anything further is to be
done on App.vue's size, it's CSS contraction (aggressive) and
template formatting (moderate) — neither of which is C2's scope.

**Recommendation: close the C2 arc with C2.3. App.vue at 500 lines
is the bounded-scope steady state.** Further line reduction would
require either non-C2 work (CSS/template contraction) or thin-
cluster extractions that don't earn their keep.

## Out of scope (explicitly)

- **Further C2.x extractions.** Per the bounded-stopping
  evaluation above. If a specific thin cluster proves to need
  extraction later (e.g., a second consumer materializes for one
  of them per ADR-0003's Port-extraction principle), do it then,
  not now.
- **CSS/template contraction** to chase ADR-0007's ≤250-line SFC
  target. Per the bounded posture and ADR-0007's "never compress
  logic to fit a budget" rule.
- **`SyncService` constructor parameter injection.** Currently
  the service reads `store` and `api` from module scope; not
  changing that contract today.
- **Tags-fetch retry on transient failure.** The current
  fail-soft `console.warn` is per ADR-0002's design.
