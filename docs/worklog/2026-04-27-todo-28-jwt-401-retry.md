# TODO #28 — JWT 401 Retry (Identity-Honest Variant) + Auth-State Bridge

- **Status:** Shipped on branch `frontend/jwt-401-retry`,
  2026-04-27. `npm run build` green; manual smoke confirmed by
  user across all four scenarios.
- **Genre:** Worklog entry — closes TODO #28 with a deviation
  from the literal spec (identity-honest re-login replaces the
  spec's `ensureAuthenticated` call) and includes a bundled
  fix for a pre-existing api-client ↔ useAuth state drift that
  the testing surfaced.
- **Date:** 2026-04-27.
- **Origin:** `docs/TODO.md` item 28 (now retired). The original
  TODO predated B5 finalization; the design here updates it for
  the post-B5 identity-honesty constraint.

## Context

`docs/TODO.md` item 28 specified silent recovery on 401: attempt
`ensureAuthenticated()` once and retry the original request,
surfacing success via `pushSystemMessage('info', ...)`. The
motivation is that 30-day JWTs cause each user to hit one
visible-failure-then-silent-recovery roughly once per month;
making the recovery transparent removes that friction.

The original spec assumed `ensureAuthenticated`'s auto-fill-as-
`local_user` was the right recovery mechanism. Post-B5 finalization
(PR #7), we know better: that auto-fill is a silent identity
substitution, exactly the failure mode the identity-aware
SyncService rework was built to close. Re-introducing it on the
401 retry path would put the data-loss bug back.

This work shipped the post-B5-aware variant: **on 401 from a
non-`/auth/*` endpoint, attempt `login(cachedUsername)` once and
retry**. Identity-preserving. For passwordless deployments the
re-login succeeds and the retry runs transparently; for
password-protected accounts the re-login fails (no password
cached) and the original 401 falls through to the existing
visible rejection flow.

During smoke testing, scenarios 2 and 4 (password account /
no cached username, mid-session 401) revealed a pre-existing
drift: api-client.ts's `request()` clears the JWT on 401 as a
side-effect of any backend call, but `useAuth.state` doesn't
update unless useAuth's own methods run. Mid-session non-`/auth/*`
401s left `auth.state` falsely `'authenticated'` while the JWT
was already gone — the auth-lifecycle UX (modal auto-open,
workspace wipe via SyncService's auth-state watcher) didn't
trigger; sync's gate stayed open, producing "spammy" follow-up
401s.

The user reframed this correctly as **architectural drift**, not
a planning gap: two physical representations of one nominal
handle ("is the user authenticated?") whose alignment is
convention-only. ADR-0005 Rule 1 (single source of truth per
nominal handle) was authored about documentation; the same risk
applies to runtime state. The PR was extended in flight to add a
callback bridge — a partial-but-honest fix that closes the gap
without restructuring the ownership.

## Approach

### `api-client.ts` — silent retry plus callback bridge

**Recursion guard:**
```typescript
private isReauthInFlight = false;
```

The inner `login()` call during retry re-enters `request()` against
`/auth/token`; the auth-endpoint check below would catch it, but
the flag is belt-and-suspenders.

**Callback bridge:**
```typescript
private onTokenInvalidatedCallback: (() => void) | null = null;
public onTokenInvalidated(cb: () => void): void {
  this.onTokenInvalidatedCallback = cb;
}
```

Invoked from the 401-throw path when a non-auth endpoint clears
the token. Auth endpoints' callers handle their own state, so
they're skipped to avoid duplicate transitions.

**`request<T>` rework:** factored the payload construction into
a `buildPayload` closure (so the retry can re-fetch with a fresh
header set reflecting the new token), then added the 401-handling
block:

```typescript
const isAuthEndpoint = path.startsWith('/auth/');
if (response.status === 401 && !this.isReauthInFlight && !isAuthEndpoint) {
  const cached = this.cachedUsername();
  if (cached) {
    this.token = null;
    this.isReauthInFlight = true;
    try {
      await this.login(cached);
      response = await fetch(`${API_BASE_URL}${path}`, buildPayload());
      if (response.ok) {
        pushSystemMessage('info', 'Session refreshed automatically.');
      }
    } catch {
      // login() or retry-fetch threw; original 401 falls through
    } finally {
      this.isReauthInFlight = false;
    }
  }
}
```

The 401 throw path now calls the callback (for non-auth endpoints
only):

```typescript
if (response.status === 401) {
  this.token = null;
  if (!isAuthEndpoint) {
    this.onTokenInvalidatedCallback?.();
  }
}
```

### `useAuth.ts` — register the callback at module init

```typescript
api.onTokenInvalidated(() => {
  if (_authState.value.kind === 'authenticated') {
    setState({ kind: 'unauthenticated' });
    pushSystemMessage('warning', 'Session expired. Please sign in again.');
  }
});
```

Module-scope so it registers exactly once. The kind-guard makes
the registration idempotent — if `_authState` is already
non-authenticated when the callback fires, no redundant
transition.

### Why `login(cached)` not `ensureAuthenticated()`

- `ensureAuthenticated` hardcodes `'local_user'` and falls back
  to register-then-login. Both behaviors silently substitute
  identity for any non-`local_user` caller — the failure mode
  closed in B5 finalization.
- `login(cached)` re-attempts the same identity that was last
  successfully authenticated. Passwordless deployments get
  silent recovery for any user; password-protected accounts
  fall through cleanly.
- The `register` fallback that `ensureAuthenticated` provides
  intentionally isn't replicated. Silent registration on
  behalf of the user is the wrong recovery for multi-tenant
  contexts. Cold-start auto-fill remains `tryAutoLogin`'s job.

## The drift observation — surfaced and filed

Mid-PR, scenarios 2 and 4 produced a behavior the user
correctly identified as architectural drift rather than a
planning gap. The api-client (predates useAuth; owns the JWT
lifecycle including the 401 token-clearing side-effect) and
`useAuth.state` (added during B1–B5; supposed to be the SPA's
source of truth for authentication identity) had drifted apart:
their alignment was conventional, not formal. Latent until B5
finalization plus auth-lifecycle-ux made `auth.state` a
load-bearing input to three watchers (SyncService gate,
SyncService workspace-wipe, UserBadge modal auto-open). Once
the consumer count grew, the drift surfaced as a live UX bug.

The callback bridge added here is a partial fix — it closes the
specific path that surfaced (api-client clears token on 401 → 
useAuth gets notified → state flips). The underlying shape —
two physical representations of one nominal handle, with
convention-only alignment — remains. The structural fix would
be either single-owner (api-client doesn't touch state at all,
just throws; useAuth or a new auth-guardian composable catches
on the way out), or a single physical representation. Both are
RFC-eligible discussions for a future date.

The pattern is filed as RFC-0001 open question 9
("Runtime-state source-of-truth coverage in Layer 1") so
audit's compliance pass picks it up. The auth-state pair is
named there as the worked example; the detection heuristic
(find pairs where module A writes state that module B claims
to own, with no formal binding) and remediation patterns
(single-owner + observers; callback bridge; collapse to one
representation, in order of structural honesty) are
documented.
