# Cluster theme — strict palette compliance (no `color-mix()`)

- **Status:** Shipped on `frontend/cluster-theme-palette-strict`,
  2026-05-05. Build green. Corrects a design-property violation
  in PR #133 that the user surfaced after merge.
- **Genre:** Bug fix — restore the cluster-theme block's
  compliance with the cluster-12 palette's design property
  (maximin perceptual distance) by removing every `color-mix()`
  call and projecting each substrate anchor onto exactly one
  palette entry.
- **Date:** 2026-05-05.

## Context

The cluster-12 palette in `palettes.css` was generated by
maximin optimization for perceptual distance — each of the 12
entries is placed as far as possible from every other in
CIELAB-ish color space. The optimization's premise is that the
12 rendered colors are categorically distinct from one another.

PR #133's `[data-theme="cluster"]` block defeated that premise
by reaching for `color-mix(in srgb, var(--cluster-12-9) 92%,
var(--cluster-12-4))` and similar interpolations to produce
"subtle" surface tiers between the canonical bg
(--cluster-12-9, pale pink) and the canonical text
(--cluster-12-4, deep purple). Those intermediate values are
not in the palette, and have smaller perceptual distance to
their two neighbours than any two original entries do — so the
maximin property no longer holds for the values actually
rendered.

The user surfaced this after seeing the merged result: "you
actually went ahead and violated the palette by mixing the
colors? Cool. Do you know how the palette was constructed?
maximin optimization for perceptual distance. Can you please
make sure to *only* use colors from the palette, no mixing?"

## What changed

### `frontend/src/assets/css/theme.css`

The entire `[data-theme="cluster"]` block was rewritten to
project each of the 16 substrate base anchors onto a single
`--cluster-12-N` entry. Every `color-mix()` call retired. New
mapping (palette entries used: 1, 2, 4, 5, 6, 9, 10, 11; unused:
0 per the user's earlier hint, 3, 7, 8 reserved for future
role-alias overrides):

| Substrate anchor    | Cluster entry     | Notes                          |
|---------------------|-------------------|--------------------------------|
| `--surface-0`       | `--cluster-12-9`  | page bg (pink)                 |
| `--surface-1`       | `--cluster-12-6`  | chrome containers (taupe)      |
| `--surface-2`       | `--cluster-12-9`  | raised → bg tone               |
| `--surface-3`       | `--cluster-12-9`  | most-raised → bg tone          |
| `--border-1`        | `--cluster-12-6`  | subtle (taupe)                 |
| `--border-2`        | `--cluster-12-6`  | default (taupe)                |
| `--border-3`        | `--cluster-12-4`  | strong / focus (purple)        |
| `--text-0`          | `--cluster-12-4`  | high emphasis (purple)         |
| `--text-1`          | `--cluster-12-4`  | body copy (purple)             |
| `--text-2`          | `--cluster-12-6`  | labels, hints (taupe)          |
| `--accent-primary`  | `--cluster-12-2`  | sky blue                       |
| `--accent-secondary`| `--cluster-12-11` | orange                         |
| `--state-success`   | `--cluster-12-1`  | bright green                   |
| `--state-warning`   | `--cluster-12-11` | orange (= accent-secondary)    |
| `--state-error`     | `--cluster-12-5`  | dark red                       |
| `--state-attention` | `--cluster-12-10` | hot pink                       |

The cluster block's docstring is now honest about why the
substrate's progressive-tier model can't be honored under
strict palette compliance: maximin optimization explicitly
prevents "close-by" colors, so subtle gradations
(`surface-0 → 1 → 2 → 3` as progressively-tinted variants of
each other) aren't possible. The substrate's role distinctions
become categorical rather than progressive — some tiers
collapse to identical values:

- **Surfaces** collapse to two effective tones: page bg (pink)
  and chrome (taupe). `--surface-1` is the chrome tier (the
  one the user has been pinning chrome containers to in recent
  commits — sidebars, tab bars, toolbars). `--surface-2` and
  `--surface-3` revert to the page bg, so elements bound to
  those tiers (raised buttons over chrome, the active tab over
  the chrome bar) appear pink-against-taupe — a visible and
  coherent inverse of the dark theme's lighter-than-bg raise
  direction.
- **Borders** collapse to two effective values: taupe (subtle
  / default) and purple (strong / focus). Inside taupe chrome
  panels, taupe borders are invisible — semantically "subtle"
  in the palette-strict sense (context-dependent visibility).
  On pink surfaces the same declarations are taupe-on-pink,
  visible.
- **Text** collapses to two effective values: deep purple
  (high-emphasis and body — the only strong-contrast text
  against both pink and taupe) and taupe (labels and hints —
  reads as "incidental" rather than "informational" against
  pink, blends into chrome on taupe).

The trade-off is documented per-tier in the block's docstring
so a future contributor doesn't reach for `color-mix()`
"because the substrate seems to want progressive gradations" —
they read the docstring first and understand why the palette
forecloses that move.

The file-header's "Available themes" section is updated
accordingly: the cluster line now names strict palette
compliance and points at the block-level docstring for the
per-tier collapse detail.

## Why no other code touched

The chart-derived helpers and role aliases at `:root` are
`var()` indirections (e.g. `--heatmap-low: var(--surface-2)`).
They resolve at use time against the active theme's base
anchors — strict palette compliance for the base anchors
means strict compliance for every alias that resolves through
them. No per-theme override needed; no other code touched.

The 8 unused palette entries (0, 3, 7, 8) are not assigned
to substrate roles. They're available for future role-alias
overrides if a specific need surfaces (e.g. a cluster-mode
override for `--heatmap-mid` to use a more vivid
visualization-side hue than what `--accent-primary` cascades
to).

## Forward notes

If the palette-strict mapping reads as too flat (some chrome
elements become invisible against neighbouring elements
because both are taupe, or both are pink), the right
correction is **not** to introduce `color-mix()` again — it's
to either (a) repin the affected element to a different
substrate tier in its SFC's `<style>` block, or (b) author
additional role aliases at `:root` that pick unused palette
entries (3 cyan-mint, 7 bright purple, 8 yellow-green) for
specific implicit roles. Both moves preserve palette
compliance.

If the substrate's category collapses (surface-0 = surface-2
= surface-3, border-1 = border-2, text-0 = text-1) become
load-bearing — i.e., the user wants chrome to render with
genuinely-distinct tiers per the dark theme's progressive
model — the principled answer is to extend the palette
itself. New cluster-N variants with intermediate values would
need to be added to `palettes.css`, with the maximin
property re-verified for the larger set. That's a palette-
authoring decision, not a substrate-side compromise.

The `useVariationPath`-shaped lesson applies here in a small
way: I conflated "the substrate's tier vocabulary" with
"freely-derivable color values" without checking whether the
palette's design property survived the derivation. The
substrate's role names are descriptors of UI semantics
(deepest, raised, subtle, default, strong); they don't
prescribe that those roles be filled by progressive tints
of a base color. Under strict palette compliance, some of
those role distinctions just aren't in the value space —
and treating them as if they were produces silent property-
violation.
