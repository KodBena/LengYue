- **Genre:** Worklog entry — frontend half of release-scope

- **Date:** 2026-04-29.

## Context

Release-scope `docs/release-scope.md` item 4 bundles two
SGF-serialisation surfaces: pass handling in the KataGo wire,
and save-to-disk for the active board's tree. The two are
genre-distinct (engine wire bug vs. UX feature) and ship
independently; this entry covers the save-to-disk half only.

Pre-existing infrastructure: `engine/sgf-writer.ts` already had
`serializeBoard(state)` and `serializeActivePath(state)` —
pure SGF serialisation. The writer iterates `node.properties`
verbatim, and the loader (via Sabaki's `data` field) populates
the full property dict per node, so round-trip is identity-on-
property-dict. Verified empirically pre-implementation:

- 4 real game records (147 / 210 / 459 / 201 nodes) round-trip
  identical
- Synthetic with passes (`W[]`), variations, comments, marks
  (LB, MA, TR, SQ, CR), root metadata round-trips identical
- Synthetic with escape sequences (`\]`, `\\` inside `C[...]`)
  round-trips identical

The "writer might predate per-node property capture" worry was
unfounded — writer reads what loader writes.

## Architectural shape

- **New composable** `src/composables/useSgfDownload.ts` (105
  lines). Exposes `downloadActiveBoard()`. Mirror of
  `useSgfLoader`'s shape: transient DOM element pattern, no
  template ref required. Fetches the active board from the
  store, calls `serializeBoard`, derives a filename from
  root-node `PB` / `PW` / `DT` properties, triggers download
  via `Blob` + transient `<a download>`. Failures push
  SystemMessage per ADR-0002.
- **Toolbar wire** `src/components/Toolbar.vue` — one new
  emit (`save-sgf`) and one new button between "Load SGF" and
  the engine connect button. Same `toolbar-btn` styling.
- **App wire** `src/App.vue` — instantiates the composable
  and forwards the emit to its `downloadActiveBoard`.

That's the entirety of the change. ~115 lines net additions.

## Filename derivation

Default filename composed from root-node SGF properties:

| `PB`        | `PW`     | `DT`         | Filename                               |
|-------------|----------|--------------|----------------------------------------|
| Lee Sedol   | AlphaGo  | 2016-03-09   | `Lee_Sedol-vs-AlphaGo-2016-03-09.sgf`  |
| (missing)   | AlphaGo  | (missing)    | `-vs-AlphaGo.sgf` *→ collapses to* `AlphaGo.sgf` (after the leading-`_` strip) |
| (missing)   | (missing)| (missing)    | `board.sgf`                            |

Sanitiser is `[a-zA-Z0-9_-]+` → `_`, plus `^_+|_+$` → ``. Cross-
platform-safe.

## What's NOT in scope

- **Pass handling in the KataGo wire** — the companion item-4
  bullet. Engine wire translation, not file-side serialisation.
  The writer round-trips passes correctly (`B[]` / `W[]`); the
  bug is between the engine and the wire layer.
- **Customisable filename / save-as dialog** — browsers don't
  expose the OS save-as picker without `<File System Access
  API>` (Chrome-only, behind a permission). The download-bar
  default is good-enough for v1.
- **Save the active variation only** — `serializeActivePath`
  exists in the writer but isn't wired. Could be a future
  "Save variation" button if there's demand; current "Save SGF"
  serialises the whole tree.
- **Round-trip persistence (re-load the saved file edits the
  board in place)** — every save creates a fresh download; the
  user is expected to re-load via "Load SGF" if they want the
  saved file back in the app.
