# Analytic Firewall — "Does the work-status SSOT manifest earn its weight, and are its load-bearing claims sound?"

**Consult date:** 2026-06-02
**Reviewer stance:** Independent. Adversarial where warranted. No stake in whether the manifest ships.
**Calibration:** Single-maintainer project; primary motive is type-sanity / code-hygiene; prizes single-source-of-truth, derivations-over-duplication, closed-vocabularies-as-types, infrastructural-excellence-over-hacks; decision-capacity is a live, named constraint. The argument is pitched at "is this the right *structure* and does it earn its weight *here*," not "is an SSOT good in the abstract."

---

## Bottom line up front

**The problem is real, but the proposed manifest is the wrong shape of fix for the part that actually failed — and its single load-bearing claim (the "generation-direction inversion") is the place it is weakest, not strongest.** The least-regret move is:

1. **Adopt G4 (the retire-on-ship checklist) now** — it is already free, already in the umbrella `CLAUDE.md`'s "Documentation is part of the work" audit, and it is the only intervention that touches the *actual* root cause of Lapse 2 (a human did not run the audit during a 24-hour proposal-to-ship window).
2. **Build the P3 *checker* — but only the contradiction half, and invert its source of truth.** The valuable, mechanizable thing here is a **lint that detects when the same work-item's status is asserted inconsistently across the three docs** (or against git for shipped-PR claims). That does **not** require a new hand-authored manifest. It requires *structured status markers in the prose the project already writes*, parsed into a transient in-memory table the same way `doc-graph` parses prose edges. Make the manifest a **cache derived from the docs**, exactly like `doc-graph.json` — *not* an authored upstream SoT. This is the inversion the note got backwards.
3. **Do not build the authored-manifest-as-SoT + P2 generated projection in v1.** It relocates the memory-dependence rather than removing it (a human must now keep the manifest true to the code *and* to the prose), it is the "hand-maintained graph becomes the stalest node" meta-irony the sibling note explicitly warned against, and its CI "consistency check" cannot do the semantic work the note implies it can.

The sharp version, in eight claims:

1. **The problem is real and correctly diagnosed.** *(verified)* Status is genuinely duplicated across three independently-editable prose docs, and a *live second instance* exists right now (the proxy pin — see claim 8). Finding 2d's "ADR-0005 Rule 1 applied to status" generalization is sound. **But** the RCA itself is precise that Lapse 2 was *status-blindness at authoring time* (Finding 2c: an author wrote "future" about a feature that shipped *yesterday*), not staleness-by-neglect. The failed guard was a human running an audit. The cheapest thing that addresses *that* is G4, which the note treats as already-subsumed but which is in fact the only piece touching the demonstrated failure surface.

2. **The "generation-direction inversion" is real as an observation but is a *liability*, not a feature.** *(analytic)* The note frames the inversion — doc-graph's manifest is *derived* (a regenerable cache), work-status's manifest is *authored* (the upstream SoT) — as the load-bearing structural fact that drives the design. It is real. But the note then treats it as neutral-or-positive when it is the single thing that should give the maintainer pause: it is an explicit admission that the work-status manifest **has none of the doc-graph's key safety property**. The doc-graph manifest *cannot* go stale relative to the docs because it is recomputed from them and CI checks the recomputation. The authored work-status manifest *can* go stale relative to both the docs and the code, because nothing computes it. The note's own sibling says this outright (`documentation-graph-artifact-plan.md` §Blind-spots: *"a hand-drawn graph would itself become the stalest node … the generator is scripted, not hand-drawn — non-negotiable"*). An authored status manifest is precisely a hand-drawn graph by another name. The inversion section is well-argued, but it argues *for the thing the sibling note forbids*.

3. **The "validator falls out for free" claim does not survive the inversion.** *(analytic, grounded in the generator read)* In `doc-graph`, the validator is free because resolution is a *computable* property: a path either points at an existing file or it does not (`extractEdges` resolves against `nodeSet`; ADR-0005's Alternative C explicitly notes resolution "is *not* soft"). The dual the note claims — "prose status disagrees with the manifest status" — is only as free as the manifest is *trustworthy*, and the manifest is hand-authored, so the check reduces to "does hand-authored-prose-A agree with hand-authored-manifest-B." That is not validation against ground truth; it is consistency between two hand-maintained artifacts, and **the contradiction it would surface is as likely to mean "the manifest is stale" as "the prose is stale."** The genuinely-free, genuinely-grounded check is the *other* one the note half-concedes and then under-weights: the **existence half** (an `open` item whose `source` path now resolves on `main`), which checks prose *against the code* — but the note correctly inherits G6's skepticism that this is a low-precision heuristic (a file existing ≠ the capability existing; a feature shipping under a different name than the entry uses is a false negative). So: the cheap-and-grounded check is heuristic, and the exact-and-hard-gate check is only as good as a second hand-maintained file. The "free validator" framing borrows the doc-graph's credibility without inheriting its mechanism.

4. **There is a cleaner design the note names as "forced, not a fork" but never actually evaluates: invert it back.** *(analytic — this is the central finding)* The note's §Design-space fork 1 dismisses doc-first-vs-manifest-first as "forced by the decision (manifest-first)." But there are *two* readings of "manifest-first," and the decision's wording (`TODO.md` *projects* from a *machine-readable* SSOT) does not actually force the authored-upstream reading. The doc-graph is *also* "machine-readable-SSOT-first" — its manifest *is* the SoT — yet it is *derived*. The design the note never seriously costs: **keep the prose docs as the authoring surface, add structured status markers to them (a small frontmatter-ish or inline-token convention), parse those into `work-status.json` as a regenerable cache, and CI-check the cache the way `doc-graph-ci` does.** This gets the project a machine-readable status SSOT (the decision's literal ask), the consistency-validator-for-free (now genuinely free, because the cache is computed), *and* avoids introducing a new hand-maintained upstream that can rot. The cost the note cites against this ("parsing free-form prose markers reliably") is exactly the cost `doc-graph` already pays and demonstrably wins — and it is *lower* here, because the project controls the marker convention going forward (it does not have to retrofit unparseable history; ADR-0005 Rule 4 already suppresses the unparseable form, as the sibling note measured). This is the same "derivations-over-duplication / SSOT" instinct the maintainer prizes, applied one level up: the authored-manifest design *creates* a fourth duplicate of status (manifest + 3 docs); the derived-cache design keeps status authored *once*, in whichever doc owns the item, and computes the rest.

5. **"`TODO.md` projects from the manifest" does not survive TODO's prose, and the note's own recommendation concedes this without naming the concession's full weight.** *(verified — TODO.md read end to end)* TODO.md is not a status table with prose footnotes; it is *prose with status markers embedded* — the analysis-tab entry runs ~17 lines of phase breakdown, the Chess-clone entry ~30 lines of backend/frontend/engine/palette decomposition, "Automatic mistake discovery" ~40 lines of open design questions, "Polymorphic chart renderer" ~40 lines of Port-surface design. The *thinking* dwarfs the *status* by an order of magnitude. The note's P2 (generate the status scaffolding between markers, hand-author the body beneath) is the only coherent generation option — P1 is correctly rejected as markdown-in-JSON — but P2's "generated header + hand-authored body" means the manifest generates a *header line and some tags*, while everything load-bearing stays hand-authored prose. At that point "TODO.md *projects from* the manifest" has collapsed to "TODO.md *has a generated status line per item*," which is barely distinguishable from P3's "TODO.md is *checked against* the manifest." The note's recommended sequencing (P3-then-P2) is honest about this — it admits P3 is "slightly weaker than the decision's wording" — but undersells it: **P2 over P3 buys almost nothing here**, because the projectable surface of TODO.md is tiny. The generated-view-in-a-sibling-`work-status.md` option (the note's lean) is cleaner precisely because it *stops pretending TODO.md is projected* and makes the generated thing a separate thin index — which is, again, just the doc-graph's own `doc-graph.md` shape, derived from a cache.

6. **The schema is mostly sound; the closed-but-amendable status vocabulary is correctly reconciled with ADR-0008; "tolerate unknown fields, not unknown statuses" is right; JSON-vs-YAML is correctly identified as mattering more here — and that last point is itself evidence against the authored design.** *(verified against ADR-0008)* The note's ADR-0008 reconciliation is genuinely correct and well-stated: a closed `statuses` set with fail-loud on an unknown member is the positive register (refuse fuzzy-fit), and "amend the enum, never fuzzy-match into it" is exactly ADR-0008's "revise the vocabulary" move; the deliberately-imprecise tags exception (`[experimental]`/`[planned]`) maps cleanly. The stable kebab-slug id is the right call and the genuinely-missing primitive (the project keyed work by mutable title or an abandoned `item N` scheme — verified in TODO.md: items 27/30c/30d/32 coexist with un-numbered items). "Forward-compatible best-effort" is coherent *once disambiguated* as the note disambiguates it (unknown *fields* tolerated, unknown *statuses* fatal) — the phrase is only self-contradictory if you conflate the two axes, and the note does not. **However:** the note flags (§Design-space fork 5, Open-question 2) that JSON's hand-edit hostility *bites here* because the manifest is hand-authored, where it never bit the doc-graph because that manifest is generated. This is correct and is a tell: the design is fighting its own substrate. In the derived-cache design (claim 4), JSON is fine again — nobody hand-edits it — and the fork dissolves. A design where the SoT-format question is genuinely hard is a design pushing against the grain.

7. **The genre boundary is mostly drawable but has one real crack, and the orthogonal-layer claim against the frontmatter plan is sound.** *(verified against `doc-graph-discipline-plan.md` and `decisions-deferred.md`)* Ruling `decisions-deferred.md` out of scope is *correct*: a deferred *decision* (chosen-against, with a revisit-trigger and an outcome-on-revisit lifecycle) is a different genre from a deferred *work item* (queued, not now) — verified by reading its entries, which are decision-records, not schedulable work. The composition claim against the frontmatter plan is also sound: that plan's `live-ledger` genre carries a *doc-level* status (`active`) and explicitly says "entries within carry their own state" — the per-*entry* layer it left open is exactly the per-*work-item* layer this manifest fills; they are orthogonal (per-document vs per-item) and neither blocks the other. **The crack** is G5's hard part — "delegate status, keep description." TODO.md's entries do not cleanly separate the two: the "Save → disconnect clears the analysis graph" entry's *status* (open) is interwoven with its *description* (the two-part fix, the registry path, the experimental-tag plan), and the RB-2/RB-3 nested sub-items carry *per-bullet* status (`~~RB-1~~ shipped`, `RB-3 closed`) inside a parent that is itself partially open. A manifest item is one `(id, status)` pair; a TODO entry can carry *three* statuses in one prose block. The note's worked example (the analysis-tab item, cleanly one status) is the easy case; the multi-status nested entries are where "one item = one status" fractures, and the note does not address them. This is survivable (model the sub-items as separate slugs) but it inflates the migration and is unacknowledged.

8. **A live second instance proves the *problem* and simultaneously undercuts the *manifest's scope*.** *(verified)* The note cites the proxy-pin drift as a fourth-axis instance: `handoff-current.md` and `CLAUDE.md` say the pin is v1.0.21, HEAD pins v1.0.27. I verified this: `git ls-tree HEAD proxy` → `b871127`, `git -C proxy describe` → **v1.0.27**, while `docs/handoff-current.md:563` and `CLAUDE.md:205` both assert **v1.0.21**. This is genuinely the same duplicated-prose-status shape — and it is excellent evidence that the *class* is real and recurring. **But it cuts against the manifest two ways the note half-sees in its Open-question 4 and then sets aside.** (a) One of the two drifted docs is `CLAUDE.md`, which the note's scope explicitly excludes (it lists only TODO / deferred / handoff). (b) The drifted handle is a *version pin* — "a fact, not a work item" — which the manifest is explicitly *not* for. So the note's own freshest, most concrete instance of the problem is **out of scope for the artifact it motivates.** That is a strong signal that the right intervention is *broader and lighter* than a work-item manifest: a general "asserted-fact-with-a-checkable-source" lint (pin value checkable against `git ls-tree`; shipped-PR checkable against git; an `open` item checkable against a source path) covers the proxy pin, the Lapse-2 case, *and* future facts the work-item schema would never model — and it needs no authored manifest at all, just the structured markers and the check.

## The core argument, expanded

### The note solves a duplication problem by adding a fourth copy

ADR-0005 Rule 1, which the note correctly invokes, says: *one owning document per handle; slimmed views delegate status to it.* The honest application of Rule 1 to status is **designate an owner and have the others delegate** — which is G5's prose form, and which requires *zero new artifacts*. The note instead designates the owner to be a *new* document (the manifest) that did not previously exist, which means every work item's status now lives in: the manifest (new SoT) + TODO.md (projected/checked) + deferred-items.md (delegating) + handoff (delegating). The note's discipline ("the others *delegate*, they don't *assert*") is the same discipline G5-as-prose would need — so the manifest does not *remove* the need for delegation discipline; it adds a fourth node that *also* needs to be kept true. The maintainer's stated preference for derivations-over-duplication should read this as suspicious: the SSOT-purist move is to compute the redundant copies, not to author a canonical one and manually sync the rest.

### Why the doc-graph precedent does *not* transfer

The note leans hard on the doc-graph as precedent ("manifest-first, committed, CI-verified-fresh, validator-for-free, single-maintainer rationale"). Every one of those properties on the doc-graph derives from a single fact: **the manifest is computed from the docs, so the docs are the SoT and the manifest cannot lie about them.** Strip that fact — which the inversion does, by construction — and the borrowed properties decay:

- "Committed, CI-verified-fresh" → on the doc-graph, CI *regenerates and diffs* (it can, because the manifest is a function of the tree). On the authored manifest, CI cannot regenerate it (there is no function to call), so "fresh" is undefined; the best it can do is "the prose doesn't contradict it," which is the weaker consistency check, not freshness.
- "Validator for free" → free on the doc-graph because resolution is computable (claim 3). Not free on the authored manifest, because the high-value check (prose-vs-manifest contradiction) compares two hand-maintained things.
- "The picture is the nicety, the validator pays rent" → on the doc-graph the validator checks docs against *ground truth* (the filesystem). On the authored manifest, the only check against ground truth is the *heuristic* existence check (G6's low-precision slice). The exact check is intra-document-set.

The note is at its most persuasive exactly where it is borrowing the sibling's hard-won credibility, and the borrow does not hold.

### What the project should build instead (the derived-cache inversion)

Keep status authored *once*, in the doc that owns each item (TODO for scheduled, deferred-items for parked, handoff for in-flight orientation). Add a small structured marker convention to the prose — the project already half-has this (`~~strikethrough~~ *(shipped 2026-05-29)*`, `[frontend]` scope tags, "Trigger:" lines). Formalize *just enough* of it to parse: a stable slug, a status token, optional `shipped: PR#NNN`, optional `source: path`. Then:

- `tools/work-status/generate.mjs` parses those markers across the three docs into a transient in-memory table (the doc-graph generator is the exact template — it already strips fenced code, resolves paths, marks dangling/ambiguous, and never silently guesses per ADR-0002).
- It writes a *derived* `docs/work-status.json` cache (committed, CI-regenerated-and-diffed for structure, exactly the existing `doc-graph-ci` shape).
- The same pass emits the **consistency report**: (a) the same slug asserted with two different statuses across docs → hard error (now genuinely computable, because both sides are parsed from the docs, not one from a hand-file); (b) a `shipped: PR#NNN` whose merge git cannot confirm, or an `open` item whose `source:` path resolves on `main` → advisory (the heuristic half, report-not-gate, inheriting G6's honest skepticism).

This delivers the decision's literal text (a machine-readable status SSOT that the human views project from), the maintainer's actual want (the three docs *cannot silently disagree*), and the type-sanity payoff (a closed status vocabulary, fail-loud on unknown members) — **without** a new hand-maintained upstream that can rot, and without the JSON-hand-edit friction the note itself flags. It is strictly closer to the doc-graph it admires.

The one thing this does *not* do is let TODO.md be "fully generated." But claim 5 showed that was never achievable anyway (the prose dwarfs the status), and the note's own P3-then-P2 sequencing already retreats to "checked-against" for v1. The derived-cache design simply *commits* to checked-against as the honest end state and stops paying for an authored SoT whose only marginal product over the checker is a generated header line.

## Direct answers to the seven evaluation questions

1. **Problem real / manifest proportionate?** Problem real (verified). Manifest *disproportionate*: it relocates memory-dependence (now the manifest must be kept true to code *and* prose) and adds a fourth status copy. The proportionate fixes are G4 (touches the demonstrated failure) + a derived-cache checker (touches the duplication structurally without a new authored SoT).

2. **Inversion sound or ornamental?** The *observation* is sound and important. The *design conclusion drawn from it* is backwards: the inversion is the reason to **not** author the manifest, because it strips exactly the safety property (auto-derivability) that makes the doc-graph trustworthy. The note presents a real weakness as a neutral structural fact.

3. **Validator free?** No. The grounded check (prose-vs-code) is the heuristic, low-precision one G6 already discounted. The exact check (prose-vs-manifest) compares two hand-maintained artifacts and is not validation against ground truth. The validator is free only under the *derived-cache* design, where both sides come from the docs.

4. **Projection survives TODO's prose?** No — verified against TODO.md's actual content. P2 collapses toward P3 because the projectable surface is a sliver of each entry. The note half-concedes this; the concession is larger than stated. The cleaner missed design is the derived cache + a thin sibling generated index, not a projected TODO.

5. **Schema sound?** Largely yes (verified). Slug-id: right and genuinely missing. Closed-but-amendable status: correctly reconciled with ADR-0008. Unknown-fields-vs-unknown-statuses: coherent. JSON-vs-YAML mattering more here: correct — and a tell that the authored design fights its substrate (the fork dissolves under the derived-cache design).

6. **Genre boundary?** `decisions-deferred.md` correctly out of scope (verified). Orthogonal-layer claim against the frontmatter plan: sound (verified). One unacknowledged crack: multi-status nested TODO entries (RB-1/2/3) break "one item = one status"; survivable as separate slugs but inflates the migration.

7. **Right next increment for a decision-capacity-limited maintainer?** The note's "checker-first, generated-view-later" staging is the right *instinct* and reduces risk. But even the checker is premature *in its authored-manifest form*, because it forces the schema-freeze + one-time migration-sweep ("1–2 focused sessions," the note's own estimate) — a real decision-and-labor load — before any value lands. The derived-cache checker front-loads less: define the marker convention, write the parser (doc-graph is the template), emit the report. G4 lands today for free. That ordering respects the named constraint.

## One concession to the note

If the maintainer's actual end-state intent is the *broader* consolidation the same deferred-items entry flags (doc retirement, a `docs/notes/` hierarchy, a mandated reorganization discipline) — and the work-status manifest is meant to be the *substrate* that later absorbs more than work-item status — then an authored SoT has a forward-looking case the narrow analysis above underweights. But the note explicitly scopes that broader arc *out* ("a separate, explicitly-deferred arc"), and the decision-capacity constraint argues against building substrate for a deferred arc. Within the note's own stated scope, the derived-cache + G4 path dominates.
