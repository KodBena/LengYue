# Frontend i18n — Contributor Guide

The frontend uses [vue-i18n](https://vue-i18n.intlify.dev/) (v11)
to surface UI strings in multiple languages. This document describes
the contributor workflow: how to add a translatable string, how to
extend the supported-locale set, and what is deliberately *not*
translated.

For the design rationale and the scoping decisions that produced
the current shape (locale roster, error-message handling, key
convention, lockstep posture), see
`/docs/notes/i18n-plan.md` in the umbrella tree. This file is the
day-to-day reference; the plan is the historical record.

## Supported locales

| Code      | Name        | Status                                             |
|-----------|-------------|----------------------------------------------------|
| `en`      | English     | Source language. Always populated.                 |
| `sv`      | Svenska     | Pilot. Maintained by the project author.           |
| `zh-CN`   | 简体中文     | Stub. Awaiting native-speaker contribution.        |
| `ja`      | 日本語       | Stub. Awaiting native-speaker contribution.        |
| `ko`      | 한국어       | Stub. Awaiting native-speaker contribution.        |

Catalogs live under `src/locales/<code>.json` as flat JSON with
dot-namespaced keys. The supported-locale registry (`SUPPORTED_LOCALES`)
is in `src/i18n/locales.ts` and is the single source of truth — the
RegistryEditor's `appearance.locale` enum reads it at render time, and
the migration's browser-detection helper consults it for fallback
matching.

## Adding a translatable string

1. **Pick a key.** Convention is `feature.subfeature.element` —
   for example, `qeubo.toolbar.applied`, `cards.modal.mint.title`,
   `sr.review.timeoutWarning`. Keys are stable English handles
   that never appear in user-facing UI; only the catalog *values*
   are user-visible.

2. **Add the key to `src/locales/en.json`.** This is the source
   catalog. Use ICU-style placeholders for interpolation:
   `"sr.session.cardCount": "{n} cards in queue"`.

3. **Add the key to every other locale's catalog.** Until a native
   speaker contributes a real translation, the value can be a
   placeholder (typically the English text prefixed with `[TODO] `,
   or just the English text — vue-i18n's missing-key warning fires
   for absent keys, not for English-mirrored ones, so the choice is
   about how loudly the contributor wants to flag the gap).

4. **Use the key in your SFC.** Two equivalent surfaces:

   ```vue
   <!-- Template (most common): -->
   <button :title="$t('cards.modal.mint.title')">
     {{ $t('cards.modal.mint.startReview') }}
   </button>
   ```

   ```ts
   // Composable / script (when the string flows through TS):
   import { useI18n } from 'vue-i18n';
   const { t } = useI18n();
   pushSystemMessage('error', t('sr.session.timeoutWarning'));
   ```

5. **Build to verify.** `npm run build` (which runs `vue-tsc -b
   && vite build`) catches catalog-shape errors at compile time.
   The dev console additionally surfaces missing-key warnings
   during `npm run dev` per the i18n plugin's `missingWarn` /
   `fallbackWarn` configuration.

## What does NOT get translated

The following are deliberately invariant under language. Translating
any of them would break protocol interop, persistence, or grep-stable
debugging. (The full inventory and rationale lives in
`/docs/notes/i18n-plan.md`.)

- **DSL symbol names** (`quality_delta`, `decisiveness`,
  `visit_ratio`, `complexity`, `winrate`, …). Referenced by string
  in palette configurations and the `analysis_env.symbols` registry;
  translating would break palette portability across locales. These
  are user-extensible identifiers, not display text.
- **Wire-shape field names** (snake_case from the backend or proxy).
  Protocol vocabulary; never user-facing.
- **Console / debug log lines.** The umbrella handoff names these
  as intentionally English; the target user is technical and benefits
  from grep-stable English logs.
- **KataGo's output** — move notation (`Q16`, `pass`), coordinate
  systems, raw win-rate / score numerics. Invariant under language.
- **File-format vocabulary** — SGF property tags (`B`, `W`, `AB`,
  `AW`, `RU`, `KM`), ruleset names where the file format defines
  them. Translating would break interop.
- **Built-in palette IDs** (`default`, `quality`, `score`, `rank`).
  User-visible palette *names* are translatable; the IDs are stable
  handles in persistence.
- **User-authored content** (card descriptions, palette overrides,
  bookmark names). Lives in the user's own locale by definition.

## Backend error messages

The backend emits English error strings; many `pushSystemMessage`
calls wrap them. The current posture: **translate the wrapper,
pass `${msg}` through in English.**

```ts
// Idiomatic:
pushSystemMessage('error', t('qeubo.verdict.failedWrapper', { msg: err.message }));

// where `qeubo.verdict.failedWrapper` resolves to:
//   en: "qEUBO verdict failed: {msg}"
//   sv: "qEUBO-bedömning misslyckades: {msg}"
```

A future backend-side dispatch could introduce structured error
codes the frontend translates from a code registry; until enough
sites accumulate to justify it, the pass-through approach ships.

## Lockstep discipline

Multi-source strings (error messages duplicated across `catch`
blocks, hardcoded fallbacks in `try` paths) need a discipline. The
current posture: **audit periodically rather than enforce at lint
time.** The pre-existing lockstep-trap pattern (PV-animation
defaults: three sources of truth that must agree) suggests this
is tractable as long as contributors flag sites at authoring
time. If duplicated error strings start drifting, revisit toward a
shared-constant or lint-rule approach.

## Adding a new locale

1. **Extend `SUPPORTED_LOCALES` in `src/i18n/locales.ts`.** Update
   `LOCALE_DISPLAY_NAMES` with the locale's name in its own script.
2. **Add a JSON catalog** at `src/locales/<code>.json` mirroring
   `en.json`'s key set (use `en.json`'s values as the working
   draft; flag entries that need native review).
3. **Register the catalog in `src/i18n/index.ts`'s `messages` map.**
4. **Update `detectBrowserLocale` in `src/i18n/locales.ts`** if
   the new locale needs a special prefix-match rule (most don't).
5. **Update this document's supported-locales table.**
6. **Update `/docs/notes/i18n-plan.md`'s status section.**

The RegistryEditor's `appearance.locale` enum picks up the new
entry automatically (it reads `SUPPORTED_LOCALES` at render time).
The schema migration that landed in version 23 backfills
existing users with their browser-detected locale, so users
whose browser already prefers the new locale will land on it
automatically the next time their workspace blob hydrates.

## Locale persistence model

The active locale lives at `store.profile.settings.appearance.locale`
in the GlobalStore. This is per-user state, persisted to the
backend via SyncService alongside the rest of `AppSettings`, so a
user's locale choice follows them across sessions and machines.

Browser detection runs *once* — during the schema 23 → 24
migration that introduced the field — to backfill existing users.
Subsequent sessions read the persisted value. New fresh installs
land on `'en'` from `defaults.ts` regardless of browser locale;
the user can flip via the RegistryEditor's `appearance.locale`
dropdown. (If new-install browser-detection turns out to matter,
add it at composable cold-start; the bias was toward predictable
defaults rather than silent locale-shifting.)
